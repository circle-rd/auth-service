### 🔴 P1 — Critique : Dashboard sessions « Recent sessions » toujours vide

**Fichiers concernés :**
- `src/routes/admin/sessions.ts`
- `frontend/src/views/admin/DashboardView.vue`

**Cause :** Double problème :

1. L'endpoint retourne `{ total, list, page, limit }` mais le dashboard attend `{ total, sessions }`. Le code `data.sessions ?? []` retourne toujours `[]` car la clé s'appelle `list`, pas `sessions`.
2. L'endpoint ne joint pas la table `user` — le dashboard attend `userName` et `userEmail` par session, mais la réponse ne contient que `userId`.

**Corrections à apporter :**
- Dans `src/routes/admin/sessions.ts` : renommer `list` → `sessions` dans la réponse ET ajouter un JOIN avec la table `user` pour récupérer `name` et `email`.
- Alternativement : mettre à jour uniquement le dashboard pour utiliser `data.list` et afficher `userId` tronqué — moins bien mais plus simple.
- Option recommandée : corriger l'endpoint (join user) + renommer le champ.

### 🟠 P2 — Important : `isMfaRequired` utilisateur non appliqué au login

**Fichiers concernés :**
- `src/auth.ts`
- `src/db/auth-schema.ts` (champ `isMfaRequired` sur la table `user`)
- `frontend/src/views/admin/UserDetailView.vue` (toggle admin)

**Cause :** `src/auth.ts` vérifie uniquement `applications.isMfaRequired` (MFA requis au niveau de l'application). Le champ `isMfaRequired` stocké sur l'utilisateur (positionnable par un admin via `PATCH /api/admin/users/:id`) n'est **jamais vérifié** pendant l'authentification. Un utilisateur marqué `isMfaRequired: true` peut se connecter sans MFA si l'application ne l'impose pas.

**Deux comportements `isMfaRequired` distincts :**
- **Sur `applications`** : force les utilisateurs d'une app à configurer le MFA avant d'accéder à l'app via OAuth → ✅ Déjà appliqué dans `customIdTokenClaims`
- **Sur `user`** : devrait forcer l'utilisateur à activer le MFA dès sa prochaine connexion, indépendamment de l'app → ❌ Non appliqué

**Correction à apporter :**
- Dans le hook `customIdTokenClaims` (ou un hook `session` BetterAuth), vérifier le champ `user.isMfaRequired` et lever une erreur `FORBIDDEN` si l'utilisateur n'a pas `twoFactorEnabled: true`.
- S'assurer que le message d'erreur guide l'utilisateur vers la page de configuration MFA.

## Nouvelle fonctionnalité : Macros de création d'application

> Demande : lors de la création d'une nouvelle application, déclencher automatiquement une série d'actions de bootstrap.

### Objectif

À la création de toute nouvelle application, le système doit automatiquement :

1. Créer deux rôles prédéfinis : `user` (rôle par défaut) et `admin`
2. Créer un abonnement `free` marqué comme par défaut (`isDefault: true`)
3. Affecter tous les utilisateurs avec le rôle global `superadmin` à l'application avec le rôle `admin` et le plan `free`
4. S'assurer que tout nouvel utilisateur ajouté à une application reçoit automatiquement le rôle par défaut (si aucun rôle n'est explicitement fourni) et le plan par défaut

**Note UX :** la création reste via la modale `AppCreateModal.vue` (pas de nouvelle route dédiée).

---

### Analyse de l'état actuel

**Ce qui existe déjà :**
- `assignDefaultPlanIfNeeded(userId, appId)` dans `src/routes/admin/applications.ts` — assigne le plan `isDefault: true` lors de l'ajout d'un utilisateur à une app ✅
- Schéma `appRoles` dans `src/db/schema.ts` — **pas de colonne `isDefault`** ❌
- La création d'app (`POST /api/admin/applications`) ne crée aucun rôle ni plan par défaut ❌
- L'ajout d'un utilisateur à une app (`POST /:id/users`) n'assigne pas de rôle par défaut ❌

    .set({ subscriptionPlanId: freePlan.id })

      eq(userApplications.userId, sa.id),
      eq(userApplications.applicationId, app.id),

### Gestion des cas limites

| Cas | Comportement |
|---|---|
| Création d'une app sans superadmin existant | Bootstrap des rôles et du plan uniquement — aucune assignation user |
| Utilisateur global `admin` (pas `superadmin`) | Non affecté automatiquement — doit être ajouté manuellement |
| Ajout d'un utilisateur avec `roleId` explicite | Ce rôle est utilisé, `assignDefaultRoleIfNeeded` n'est PAS appelé |
| Ajout d'un utilisateur sans `roleId` | `assignDefaultRoleIfNeeded` est appelé — si un rôle par défaut existe, il est assigné |
| Rôle par défaut supprimé | Autorisé (sous réserve qu'aucun user n'y soit affecté) — les futurs nouveaux users n'auront pas de rôle auto-assigné jusqu'à ce qu'un nouveau défaut soit défini |
| Deux rôles `isDefault: true` en concurrence | Impossible — l'API reset les autres avant de SET le nouveau |
| App existante (avant cette feature) | Non affectée — pas de rétroactivité |
