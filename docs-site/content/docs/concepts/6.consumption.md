# Consumption tracking

Auth Service provides a lightweight **consumption accounting API** for recording arbitrary numeric counters per user per application. Common use cases: API call counts, credit balances, storage usage, or token spend.

## Data model

### Consumption entries

Every call to `POST /api/consumption` inserts a raw entry:

| Field | Type | Description |
|---|---|---|
| `id` | UUID | Primary key |
| `userId` | string | User identifier |
| `applicationId` | UUID | Application identifier |
| `key` | string | Counter name, e.g. `api.calls`, `tokens.used` |
| `value` | numeric | Numeric delta (positive or negative) |
| `createdAt` | timestamp | Record creation time |

### Consumption aggregates

In the same transaction, an **upsert** maintains a running total:

| Field | Type | Description |
|---|---|---|
| `userId` | string | User identifier |
| `applicationId` | UUID | Application identifier |
| `key` | string | Counter name |
| `total` | numeric | Running total (sum of all entries) |
| `updatedAt` | timestamp | Last update time |

The aggregate supports negative values — credits can be debited by recording a negative `value`.

## Key format

Keys must match `/^[a-zA-Z0-9.]+$/` (alphanumeric with dots). Examples: `api.calls`, `tokens.used`, `storage.gb`.

## Authentication

The consumption endpoint accepts two forms of authentication:

1. **Bearer JWT** — an access token issued via the `client_credentials` grant. The token is verified by BetterAuth's `verifyAccessToken` API.
2. **Admin session cookie** — a valid session for a user with the `admin` or `superadmin` global role.

All other callers receive `403 CONS_004`.

## API endpoints

### Record consumption

```
POST /api/consumption
Authorization: Bearer <access_token>
```

```json
{
  "userId": "<user_id>",
  "applicationId": "<app_uuid>",
  "key": "api.calls",
  "value": 1
}
```

The `userId` and `applicationId` pair must correspond to an existing `userApplications` row (`CONS_003` otherwise).

Response:

```json
{
  "success": true,
  "aggregate": {
    "key": "api.calls",
    "total": "42"
  }
}
```

### Get aggregates for a user + app

```
GET /api/consumption/:userId/:applicationId
```

Returns all aggregate counters for the given user/application pair:

```json
{
  "aggregates": [
    { "key": "api.calls", "total": "42", "updatedAt": "..." },
    { "key": "tokens.used", "total": "1500", "updatedAt": "..." }
  ]
}
```

### Reset a counter

```
DELETE /api/consumption/:userId/:applicationId/:key
```

Removes the aggregate row and all raw entry rows for the given key. Admin session required.

## Admin consumption endpoints

Superadmin users can also access consumption data across all users via:

```
GET /api/admin/consumption/:applicationId
```

Returns all aggregates for all users of the given application, with pagination support.

## Backend service pattern

A backend service using the `client_credentials` grant to record consumption:

```ts
// 1. Obtain an access token (cache this and refresh before expiry)
const tokenRes = await fetch("https://auth.example.com/api/auth/oauth2/token", {
  method: "POST",
  headers: { "Content-Type": "application/x-www-form-urlencoded" },
  body: new URLSearchParams({
    grant_type: "client_credentials",
    client_id: "my-app",
    client_secret: process.env.APP_CLIENT_SECRET!,
    scope: "openid",
    resource: "https://api.example.com",
  }),
});
const { access_token } = await tokenRes.json();

// 2. Record a consumption event
await fetch("https://auth.example.com/api/consumption", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    Authorization: `Bearer ${access_token}`,
  },
  body: JSON.stringify({
    userId: req.user.sub,
    applicationId: "app-uuid",
    key: "api.calls",
    value: 1,
  }),
});
```
