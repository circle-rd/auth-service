# Configuration

All configuration is loaded from environment variables at startup. The schema is validated by [Zod](https://zod.dev/) in `src/config.ts`; the process exits with a clear error message if any required variable is missing or malformed.

## Core variables

| Variable | Required | Default | Description |
|---|---|---|---|
| `PORT` | no | `3001` | TCP port the Fastify server binds to |
| `HOST` | no | `0.0.0.0` | Bind address |
| `NODE_ENV` | no | `development` | `development`, `production`, or `test` |
| `BETTER_AUTH_SECRET` | **yes** | — | 32-byte random secret — `openssl rand -base64 32` |
| `BETTER_AUTH_URL` | **yes** | — | Public base URL of the service, e.g. `https://auth.example.com` |
| `DATABASE_URL` | **yes** | — | PostgreSQL connection URL |

## Bootstrap

| Variable | Required | Description |
|---|---|---|
| `ADMIN_EMAIL` | no | Email of the superadmin created at first boot |
| `ADMIN_PASSWORD` | no | Password for the superadmin (minimum 8 characters) |

When both variables are set and no superadmin exists yet, `src/bootstrap.ts` creates the account automatically on server startup. Subsequent starts are no-ops.

## CORS and sessions

| Variable | Required | Default | Description |
|---|---|---|---|
| `CORS_ORIGINS` | no | `http://localhost:5173` | Comma-separated list of allowed origins |
| `SESSION_DOMAIN` | no | — | Top-level domain for cross-subdomain cookies, e.g. `example.com` |

When `SESSION_DOMAIN` is set, BetterAuth enables `crossSubDomainCookies` so that a session cookie set on `auth.example.com` is readable on `app.example.com`.

## Email (SMTP)

All SMTP variables are optional. When `SMTP_HOST` is not set, password reset and email verification emails are silently skipped with a console warning.

| Variable | Default | Description |
|---|---|---|
| `SMTP_HOST` | — | SMTP server hostname |
| `SMTP_PORT` | `587` | SMTP port |
| `SMTP_USER` | — | SMTP username |
| `SMTP_PASS` | — | SMTP password |
| `SMTP_FROM` | `auth-service <no-reply@localhost>` | Sender address for outgoing emails |

## OAuth provider

| Variable | Required | Description |
|---|---|---|
| `OAUTH_VALID_AUDIENCES` | no | Comma-separated list of valid `aud` values for JWT access tokens (RFC 8707). When set, clients must include `resource=<url>` in authorization requests to receive a JWT instead of an opaque token. Example: `https://api.example.com,https://mcp.example.com` |

## Social login providers

Each provider requires both `CLIENT_ID` and `CLIENT_SECRET` to be enabled. Providing only one of the two keeps the provider disabled.

| Provider | Variables |
|---|---|
| Google | `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` |
| GitHub | `GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET` |
| LinkedIn | `LINKEDIN_CLIENT_ID`, `LINKEDIN_CLIENT_SECRET` |
| Microsoft | `MICROSOFT_CLIENT_ID`, `MICROSOFT_CLIENT_SECRET` |
| Apple | `APPLE_CLIENT_ID`, `APPLE_CLIENT_SECRET` |

## Stripe (optional billing)

| Variable | Description |
|---|---|
| `STRIPE_SECRET_KEY` | Stripe secret API key. When set, creating a subscription plan automatically creates a matching Stripe Product. |
| `STRIPE_WEBHOOK_SECRET` | Stripe webhook signing secret for the `/api/webhooks/stripe` endpoint. |

## Custom templates

| Variable | Description |
|---|---|
| `TEMPLATES_DIR` | Path to a directory containing custom HTML templates for the login, register, verify-email, and select-org pages. See [Custom templates](/docs/guides/custom-templates). |
