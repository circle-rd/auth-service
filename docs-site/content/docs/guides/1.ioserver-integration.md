# Integrating with IOServer OIDC

Auth Service is designed to work out of the box with [IOServer OIDC](https://docs.circle-cyber.com/ioserver-oidc/), a drop-in JWT middleware set for IOServer applications. No secrets are shared between the two services — the backend verifies tokens using the public JWKS endpoint.

## Prerequisites

- A running auth-service instance with a public `BETTER_AUTH_URL`.
- An IOServer backend with `ioserver-oidc` installed.
- An application registered in auth-service with an appropriate OAuth client configuration.

## Application setup in auth-service

Register a confidential application for your IOServer backend:

1. Log in to the auth-service admin SPA.
2. Create an application (`slug: my-api`, `isPublic: false`).
3. Add the scopes your frontend will request: `openid`, `profile`, `email`, `roles`, `features`.
4. Note the client secret returned at creation time.

For frontend applications (SPAs), create a **public** application (`isPublic: true`). Public clients use PKCE and do not require a secret.

## Frontend — authorize and obtain tokens

```ts
// Example using the @better-auth/client package or any OAuth 2.0 PKCE client library.
// The auth-service authorization endpoint:
const authorizeUrl = "https://auth.example.com/api/auth/oauth2/authorize";

const params = new URLSearchParams({
  client_id: "my-frontend",       // application slug
  response_type: "code",
  redirect_uri: "https://app.example.com/callback",
  scope: "openid profile email roles features",
  resource: "https://api.example.com",  // must match OAUTH_VALID_AUDIENCES
  code_challenge: await generateCodeChallenge(verifier),
  code_challenge_method: "S256",
});

window.location.href = `${authorizeUrl}?${params}`;
```

## Backend — verify tokens via JWKS

```ts
import { IOServer } from "@x42en/ioserver";
import { OidcConfigManager, OidcHttpMiddleware } from "ioserver-oidc";

const server = new IOServer({ ... });

const oidc = new OidcConfigManager({
  authServerUrl: "https://auth.example.com",
  audience: "https://api.example.com",
});
await oidc.init();

server.addMiddleware({
  name: "oidc",
  middleware: new OidcHttpMiddleware(oidc, appHandle),
});

server.start();
```

The middleware fetches the JWKS from `https://auth.example.com/api/auth/jwks`, caches the keys, and verifies the `aud` and `iss` claims on every request.

## Reading claims in a controller

```ts
class DocumentsController {
  async getDocuments(req: Request) {
    const user = req.user; // injected by OidcHttpMiddleware
    const roles: string[] = user.roles ?? [];
    const features: Record<string, unknown> = user.features ?? {};

    if (!roles.includes("editor")) {
      throw new Error("Forbidden");
    }

    // gate on a feature flag
    if (!features.apiAccess) {
      throw new Error("API access not available on your plan");
    }

    return db.documents.findMany({ where: { orgId: user.org_id } });
  }
}
```

## OIDC discovery

All configuration parameters can be auto-discovered from:

```
GET https://auth.example.com/api/auth/.well-known/openid-configuration
```

Client libraries that support OIDC discovery will configure themselves automatically from this URL.
