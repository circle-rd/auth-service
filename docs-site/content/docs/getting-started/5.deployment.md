# Deployment

## Docker image

Auth Service uses a **three-stage Docker build**:

1. **Stage 1 — frontend builder** — installs Vue SPA dependencies and runs `pnpm build` inside `frontend/`.
2. **Stage 2 — backend builder** — installs all dependencies and compiles TypeScript to `dist/`.
3. **Stage 3 — runtime** — production-only dependencies, compiled backend, built frontend, and Drizzle migration files.

The final image is based on `node:22-alpine` and exposes port **3001**.

## Build and run

```sh
# Copy and edit your production .env
cp .env.example .env

docker compose build
docker compose up -d
```

The `auth-service` container depends on `postgres` with a health check, so it starts only after PostgreSQL is ready to accept connections.

**Database migrations run automatically** at container startup via `runMigrations()` in `src/migrate.ts`. No manual migration step is required in production.

## Production environment variables

The minimum set for a production deployment:

```dotenv
PORT=3001
HOST=0.0.0.0
NODE_ENV=production

BETTER_AUTH_SECRET=<openssl rand -base64 32>
BETTER_AUTH_URL=https://auth.example.com

DATABASE_URL=postgres://auth:strongpassword@postgres:5432/auth

POSTGRES_USER=auth
POSTGRES_PASSWORD=strongpassword
POSTGRES_DB=auth

ADMIN_EMAIL=admin@example.com
ADMIN_PASSWORD=strongpassword

CORS_ORIGINS=https://app.example.com
```

## Reverse proxy

Auth Service is designed to run behind a reverse proxy for TLS termination. A minimal Traefik label configuration:

```yaml
labels:
  - traefik.enable=true
  - traefik.http.routers.auth-service.rule=Host(`auth.example.com`)
  - traefik.http.routers.auth-service.entrypoints=websecure
  - traefik.http.routers.auth-service.tls.certresolver=letsencrypt
  - traefik.http.services.auth-service.loadbalancer.server.port=3001
```

`BETTER_AUTH_URL` must match the public hostname (`https://auth.example.com` in the example above) — BetterAuth uses this value as the `issuer` in discovery documents and ID tokens.

## Health check

```sh
curl https://auth.example.com/health
# → {"status":"ok","version":"0.1.0"}
```

The Docker Compose file does not define a `HEALTHCHECK` instruction — add one if your orchestrator requires it:

```yaml
healthcheck:
  test: ["CMD-SHELL", "wget -qO /dev/null http://localhost:3001/health || exit 1"]
  interval: 30s
  timeout: 5s
  start_period: 20s
  retries: 3
```

## Database migrations (standalone)

In environments where you want to run migrations separately (e.g. a Kubernetes init container):

```sh
pnpm db:migrate
```

For schema-only changes during development (no migration files generated):

```sh
pnpm db:push
```
