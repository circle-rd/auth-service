# RBAC — Roles and permissions

Auth Service implements **per-application RBAC**. Each application has its own isolated set of roles and permissions. A user can hold different roles in different applications.

## Model

```
Application
  └── Role (e.g. "admin", "editor", "viewer")
        └── Permission (resource + action, e.g. "documents.write")

User ── UserAppRole ──► Role (scoped to Application)
```

### Roles

| Field | Type | Description |
|---|---|---|
| `id` | UUID | Primary key |
| `applicationId` | UUID | Application this role belongs to |
| `name` | string | Role name (unique per application, max 64 chars) |
| `description` | string | Optional description |
| `isDefault` | boolean | Automatically assigned to new users on first access |

Only one role per application can have `isDefault = true`. Creating a new default role resets the existing default.

### Permissions

| Field | Type | Description |
|---|---|---|
| `id` | UUID | Primary key |
| `applicationId` | UUID | Application this permission belongs to |
| `resource` | string | Resource name, e.g. `documents`, `reports` |
| `action` | `"read"` \| `"write"` | Action on the resource |

Permission strings in the JWT follow this convention:
- `read` action → `resource` (e.g. `documents`)
- `write` action → `resource.write` (e.g. `documents.write`)

This avoids redundancy: a role with `documents.write` implicitly covers `documents`.

## Admin API endpoints

All role and permission endpoints require an `admin` or `superadmin` session.

### List roles with permissions

```
GET /api/admin/applications/:appId/roles
```

Returns all roles for the application, each with a `permissionIds` array.

### Create a role

```
POST /api/admin/applications/:appId/roles
```

```json
{
  "name": "editor",
  "description": "Can read and write documents",
  "isDefault": false
}
```

### Update a role

```
PATCH /api/admin/applications/:appId/roles/:roleId
```

### Delete a role

```
DELETE /api/admin/applications/:appId/roles/:roleId
```

Blocked if the role is currently assigned to active users (`PERM_006`).

### Assign permissions to a role

```
POST /api/admin/applications/:appId/roles/:roleId/permissions
```

```json
{
  "permissionIds": ["uuid-1", "uuid-2"]
}
```

This is a replace operation — the existing permission set is replaced atomically.

### List permissions

```
GET /api/admin/applications/:appId/permissions
```

### Create a permission

```
POST /api/admin/applications/:appId/permissions
```

```json
{
  "resource": "documents",
  "action": "write"
}
```

Permission names must match `/^[a-zA-Z][a-zA-Z0-9._-]*$/`.

### Delete a permission

```
DELETE /api/admin/applications/:appId/permissions/:permissionId
```

## JWT claims

When a client requests the `roles` and/or `permissions` scopes, the access token includes:

```json
{
  "roles": ["editor"],
  "permissions": ["documents", "documents.write", "reports"]
}
```

Claims are populated by `getUserClaims()` in `src/services/claims.ts`. If the user has no role in the application, both arrays are empty.

## Global user roles

In addition to per-application roles, every user has a **global system role** managed by BetterAuth's `admin` plugin:

| Role | Capabilities |
|---|---|
| `user` | Default. No admin API access. |
| `admin` | Can manage users, sessions, and applications via the admin API. Can impersonate non-admin users. |
| `superadmin` | Full admin privileges including impersonating admin users. Cannot be deleted if it is the last superadmin. |

Global roles are set via `PATCH /api/admin/users/:id` and are stored in BetterAuth's `user` table.
