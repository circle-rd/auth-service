# OAuth 2.1 / OIDC endpoints

Auth Service exposes a standards-compliant OAuth 2.1 / OIDC authorization server powered by `@better-auth/oauth-provider`. All OAuth and OIDC endpoints are served under the `/api/auth/` prefix.

## Discovery

### OIDC discovery document

```
GET /api/auth/.well-known/openid-configuration
```

Returns the OIDC Provider Metadata document (RFC 8414 / OIDC Core §4). Clients and libraries should auto-configure themselves from this endpoint.

### Authorization server metadata (RFC 8414)

```
GET /api/auth/.well-known/oauth-authorization-server
```

Equivalent OAuth 2.0 authorization server metadata document.

### JSON Web Key Set

```
GET /api/auth/jwks
```

The public key set used to verify JWT access tokens. Keys are RS256 or ES256 depending on BetterAuth's JWT plugin configuration.

## Core endpoints

### Authorization endpoint

```
GET/POST /api/auth/oauth2/authorize
```

Initiates the authorization code flow. Required parameters:

| Parameter | Description |
|---|---|
| `client_id` | Application slug |
| `response_type` | Must be `code` |
| `redirect_uri` | Must match one of the application's `redirectUris` |
| `scope` | Space-separated scope list |
| `code_challenge` | PKCE code challenge (required for public clients) |
| `code_challenge_method` | Must be `S256` |
| `resource` | RFC 8707 resource indicator — set to a configured audience URL to receive a JWT |

When `resource` is included and matches a value in `OAUTH_VALID_AUDIENCES`, the issued access token is a signed JWT containing custom claims. Without `resource`, an opaque token is issued.

After authentication, BetterAuth redirects to `/login` (the login page) carrying the complete signed OAuth parameters in the query string.

### Token endpoint

```
POST /api/auth/oauth2/token
```

Exchanges an authorization code or refresh token for access/ID tokens.

**Authorization code grant:**

```http
POST /api/auth/oauth2/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=<authorization_code>
&redirect_uri=<redirect_uri>
&client_id=<slug>
&client_secret=<secret>       # confidential clients
&code_verifier=<verifier>     # public clients (PKCE)
```

**Client credentials grant:**

```http
POST /api/auth/oauth2/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=<slug>
&client_secret=<secret>
&scope=openid
&resource=<audience_url>
```

### UserInfo endpoint

```
GET /api/auth/oauth2/userinfo
Authorization: Bearer <access_token>
```

Returns claims about the authenticated user. The response includes the scopes requested during authorization.

## Supported scopes

| Scope | Claims added to token |
|---|---|
| `openid` | Required for OIDC. Adds `sub`. |
| `profile` | `name`, `company` |
| `email` | `email` |
| `offline_access` | Issues a refresh token |
| `roles` | `roles` — array of the user's role names in the application |
| `permissions` | `permissions` — array of permission strings |
| `features` | `features` — JSON object from the user's subscription plan |
| `org` | `org_id` — the user's active organization ID |

## JWT token structure

When `resource` is included in the authorization request and matches a configured audience, the access token is a signed JWT:

```json
{
  "iss": "https://auth.example.com",
  "sub": "<user_id>",
  "aud": "https://api.example.com",
  "exp": 1234567890,
  "iat": 1234567890,
  "email": "user@example.com",
  "name": "Alice Smith",
  "roles": ["editor"],
  "permissions": ["documents", "documents.write"],
  "features": { "maxProjects": 50, "apiAccess": true },
  "org_id": "org_01hwxyz..."
}
```

## Consent page

When `skipConsent` is `false` for an application, BetterAuth redirects to `/oauth2/consent` after authentication. The Vue admin SPA handles this route when the embedded frontend is present.

## Login pages

BetterAuth redirects to `/login` after calling the authorization endpoint. Auth Service serves either:

1. A custom HTML template from `TEMPLATES_DIR/<appSlug>/login.html`
2. A custom HTML template from `TEMPLATES_DIR/default/login.html`
3. The built-in template shipped with auth-service
4. The Vue SPA `index.html` (when no template matches)

The login page receives the full OAuth query string so the authorization flow can be resumed after authentication.
