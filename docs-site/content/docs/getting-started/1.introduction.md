# Introduction

Auth Service is a self-hosted **OAuth 2.1 / OIDC identity provider** built on [BetterAuth](https://better-auth.com) v1.5.6. It acts as the central authentication and authorisation hub for a fleet of backend services and frontend applications.

## What it provides

- **OAuth 2.1 authorization server** — authorization code + PKCE, client credentials, token refresh, token introspection, JWKS endpoint, and OIDC discovery.
- **Multi-application model** — each registered application is an independent OAuth 2.0 client with its own RBAC model, subscription plans, and access policy.
- **Per-app RBAC** — roles and permissions are defined per application and injected as `roles` / `permissions` claims in JWT access tokens.
- **MFA** — TOTP (authenticator apps) and FIDO2 passkeys (YubiKey, Touch ID, Windows Hello) via `@better-auth/passkey`.
- **Subscription plans** — plans carry a `features` JSON object that is surfaced as a `features` claim in tokens. Optional Stripe integration for billing.
- **Consumption tracking** — a lightweight accounting layer for recording and querying numeric counters (API calls, credits, storage) per user per application.
- **Organizations** — multi-tenant support; users belong to organizations and can carry an `org_id` claim in their access tokens.
- **Embedded admin SPA** — a Vue 3 frontend (served at `/`) provides a full management UI for users, applications, organizations, roles, plans, and sessions.

## When to use auth-service

Auth Service is the right choice when you need:

- A private OAuth 2.1 / OIDC provider with no dependency on a cloud identity service.
- Fine-grained per-application access control that travels inside tokens, so downstream services do not need to query the auth provider on every request.
- A single place to manage users across multiple internal applications (REST APIs, MCP servers, frontends) with different role models.

It is **not** a drop-in replacement for a full IAM platform such as Keycloak. It prioritises simplicity and tight integration with the CIRCLE Cyber stack over broad protocol coverage.

## How it fits in

```
Browser / CLI / MCP client
        │
        │  OAuth 2.1 authorization code + PKCE
        ▼
┌─────────────────────┐
│    auth-service     │  ← this project
│  (BetterAuth core)  │
└────────┬────────────┘
         │  JWT access token (roles, permissions, features, org_id)
         ▼
┌─────────────────────┐
│  backend service    │  e.g. IOServer + IOServer-OIDC middleware
│  (resource server)  │  verifies token against JWKS, no secret required
└─────────────────────┘
```

The backend service uses [IOServer OIDC](https://docs.circle-cyber.com/ioserver-oidc/) (or any JWKS-capable middleware) to verify tokens issued by auth-service. No shared secret is required — validation is done with the public key set published at `/api/auth/jwks`.

## Stack

| Layer | Technology |
|---|---|
| Runtime | Node.js 22, TypeScript (ESM) |
| Web framework | Fastify 5 |
| Auth engine | BetterAuth 1.5.6 + `@better-auth/oauth-provider` |
| ORM | Drizzle ORM + `postgres` driver |
| Database | PostgreSQL 17 |
| Frontend | Vue 3 + Vite + Tailwind CSS v4 |
| Container | Docker (multi-stage, Node 22 Alpine) |
